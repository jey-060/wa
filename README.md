기술 스택 선택 이유
프론트엔드는 HTML, CSS, JavaScript를 사용하였다. 별도의 프레임워크 없이 구현함으로써 브라우저에서 요청이 발생하고, 서버와 통신한 뒤 그 결과가 화면에 반영되는 과정을 직접 확인할 수 있었다.

백엔드는 Node.js와 Express를 사용하였다. Express는 구조가 비교적 단순하여 라우팅, 요청 처리, 응답 반환과 같은 서버의 기본 동작을 이해하기에 적합하다고 판단하였다.

데이터베이스는 PostgreSQL을 사용하였다. 회원 정보를 저장하고 조회하는 기능을 구현하면서 서버와 데이터베이스가 실제로 어떻게 연결되고 상호작용하는지 확인할 수 있었고, SQL을 직접 사용해보며 데이터 처리 과정을 이해할 수 있었다.

클라이언트와 서버 간 데이터는 JSON 형식으로 주고받았다. JSON은 웹 환경에서 널리 사용되는 방식이기 때문에 요청과 응답 구조를 확인하기에도 적합하였다.

배포는 Render와 AWS Lightsail을 함께 사용하였다. Render에는 Node.js 서버를 배포하고, Lightsail에는 Nginx를 설치하여 리버스 프록시 구조를 구성하였다. 이를 통해 단순히 서버를 배포하는 데서 끝나는 것이 아니라, 실제 서비스 환경에서 요청이 어떤 경로로 전달되는지까지 학습할 수 있었다.

전체 시스템 구조
전체 구조는 다음과 같다.

Client(브라우저) → Nginx(AWS Lightsail) → Node.js 서버(Render) → PostgreSQL(Render)

사용자가 도메인으로 접속하면 요청은 먼저 AWS Lightsail의 Nginx로 들어온다. Nginx는 리버스 프록시 역할을 하며 HTTPS 연결을 처리한 뒤, 요청을 Render에 배포된 Node.js 서버로 전달한다. 백엔드 서버는 필요한 작업을 수행하면서 PostgreSQL과 통신하고, 처리 결과를 다시 클라이언트에 응답한다.

구현 기능
회원가입 기능에서는 사용자가 입력한 아이디와 비밀번호를 데이터베이스에 저장하였다. 비밀번호는 평문으로 저장하지 않고 bcrypt로 해시한 뒤 저장하였다. 또한 아이디에는 영문과 숫자만 사용할 수 있도록 제한하였고, 비밀번호에는 특수문자를 1개 이상 포함하도록 검사하였다.

로그인 기능에서는 입력한 아이디로 사용자 정보를 조회한 뒤, bcrypt를 이용해 비밀번호를 검증하여 로그인 여부를 판단하였다. 로그인 과정에서도 아이디 형식을 확인하여 잘못된 입력을 미리 걸러내도록 하였다.

프로필 조회 기능은 로그인한 사용자만 접근할 수 있도록 구현하였다. 인증은 세션 기반으로 처리하였으며, 로그인 성공 시 서버가 세션을 생성하고 브라우저에는 세션 ID가 담긴 쿠키가 저장되도록 구성하였다. 이후 서버는 이 세션 ID를 바탕으로 사용자의 로그인 상태를 확인한다.

또한 SQL Injection을 방지하기 위해 사용자 입력값을 쿼리 문자열에 직접 이어붙이지 않고, Prepared Statement 방식으로 쿼리와 데이터를 분리하였다.

구현 과정 및 학습 방식
처음에는 AI가 생성한 코드를 기반으로 전체 구조를 먼저 실행해본 뒤, 각 코드가 어떤 역할을 하는지 분석하는 방식으로 학습을 진행하였다. 웹 서비스가 브라우저, 서버, 데이터베이스 사이에서 어떤 흐름으로 동작하는지를 먼저 확인하고, 이후 코드를 하나씩 이해하는 방식이 더 효과적이라고 느꼈다.

또한 일부러 설정 파일이나 코드 일부를 수정해 오류를 발생시켜 보고, 그 원인을 직접 찾는 과정도 반복하였다. 예를 들어 Nginx 설정 충돌, DNS 연결 문제, 세션 설정 문제, 프론트엔드 JavaScript 문법 오류 등을 직접 해결하면서 단순히 기능을 구현하는 것보다 훨씬 많은 것을 배울 수 있었다.

특히 브라우저 개발자 도구의 Console과 Network 탭, 서버 로그, Render 배포 로그 등을 함께 확인하는 습관이 중요하다는 점을 알게 되었다. 이 과정을 통해 웹 서비스는 단순히 코드만 작성한다고 끝나는 것이 아니라, 실제 요청과 응답의 흐름을 함께 보면서 점검해야 한다는 점을 체감할 수 있었다.

배포 과정
먼저 로컬 환경에서 Node.js 서버를 실행하여 회원가입, 로그인, 프로필 조회 기능이 정상 동작하는지 확인하였다. 이후 Render를 이용해 백엔드 서버를 배포하였다.

그다음 AWS Lightsail에 Ubuntu 서버를 생성하고, Nginx를 설치하여 리버스 프록시를 설정하였다. 이후 도메인을 구매해 DNS를 연결하고, 마지막으로 Certbot을 이용해 HTTPS를 적용하였다.

문제 해결 과정
배포 과정에서는 예상보다 다양한 문제가 발생하였다.

먼저 Nginx 설정 파일에 문법 오류가 있어 서버가 재시작되지 않는 문제가 있었다. 이 문제는 nginx -t 명령어로 설정 파일을 검사하면서 해결하였다. 이를 통해 설정 파일을 수정한 뒤에는 바로 재시작하기보다 먼저 문법 검사를 해보는 습관이 중요하다는 점을 알게 되었다.

또 한 번은 502 Bad Gateway 오류가 발생하였다. 처음에는 백엔드 서버 자체의 문제라고 생각했지만, 확인해보니 Nginx에서 요청을 넘길 때 필요한 프록시 헤더 설정이 부족했던 것이 원인이었다. proxy_set_header 관련 설정을 추가한 뒤 정상적으로 연결할 수 있었다. 이 과정을 통해 리버스 프록시는 단순히 요청만 전달하는 것이 아니라, 어떤 요청 정보를 함께 넘기느냐도 중요하다는 점을 배웠다.

HTTPS 적용 과정에서는 www 도메인 DNS 설정과 도메인이 실제 서버를 가리키는지 여부를 함께 확인해야 했다. Namecheap DNS 설정과 AWS Lightsail의 고정 IP 연결 상태를 점검하면서, 최종적으로 루트 도메인과 www 도메인이 모두 같은 서버를 가리키도록 수정하였다.

또 AWS Lightsail 방화벽에서 HTTP와 HTTPS 포트가 열려 있는지도 함께 확인하였다. 서버 내부에서는 Nginx가 정상적으로 실행 중이더라도, 외부에서 접속이 되지 않는 경우가 있었기 때문이다. 이를 통해 서버 설정뿐 아니라 네트워크 설정도 함께 점검해야 한다는 점을 알게 되었다.

배포 후에는 브라우저 주소가 onrender.com 도메인으로 바뀌는 문제도 있었다. 처음에는 서버 설정 문제로 생각했지만, 실제로는 브라우저가 이전 리다이렉트 응답을 캐시하고 있었던 부분도 함께 작용하고 있었다. 개발자 도구의 Network 탭에서 실제 응답 상태를 확인하고, 캐시를 비운 상태에서 다시 요청 흐름을 점검하면서 문제를 해결하였다. 이 과정을 통해 배포 환경에서는 서버 설정뿐 아니라 브라우저 캐시도 함께 확인해야 한다는 점을 배울 수 있었다.

회원가입과 로그인 기능을 점검하는 과정에서는 프론트엔드와 백엔드의 연결도 다시 확인하였다. 회원가입이 정상적으로 동작하지 않았던 원인을 추적하면서, HTML과 JavaScript의 문법 오류뿐 아니라 서버가 반환하는 응답과 브라우저의 요청 흐름도 함께 살펴보게 되었다. 특히 브라우저 개발자 도구의 Console과 Network 탭을 이용해 요청이 실제로 전송되는지, 서버가 어떤 상태 코드와 응답을 반환하는지를 확인하면서 문제를 해결하였다.

또한 데이터베이스 관리 과정도 직접 확인하였다. Render에서 PostgreSQL 데이터베이스를 별도로 생성하여 사용하였으며, 웹 서비스가 DATABASE_URL 환경 변수를 통해 이 데이터베이스와 연결된다는 점을 확인하였다. 테스트 과정에서는 특정 아이디가 이미 존재해 회원가입이 실패하는 경우도 있었고, 이 과정에서 데이터베이스 내부 데이터를 직접 조회하거나 삭제하는 방법도 살펴보았다. pgAdmin과 같은 도구를 이용해 외부에서 데이터베이스에 접속하고, users 테이블의 데이터를 조회하거나 삭제하면서 서비스 데이터가 실제로 어떻게 저장되는지 확인할 수 있었다.

추가로 Render PostgreSQL의 계정 정보 관리 방식도 확인하였다. 단순히 비밀번호를 직접 수정하는 방식이 아니라, 새로운 credential을 발급하고 그에 맞게 웹 서비스의 DATABASE_URL도 함께 갱신해야 한다는 점을 알게 되었다. 이를 통해 배포 이후에도 서버와 데이터베이스의 연결 정보를 함께 관리해야 한다는 점을 이해할 수 있었다.
